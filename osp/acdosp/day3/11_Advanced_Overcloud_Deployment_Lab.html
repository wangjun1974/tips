In this lab you implement an overcloud with the following features:

  • Keystone integration with Red Hat Identity Manager (IdM)

  • TLS everywhere (TLS on public and internal overcloud endpoints)

  • Custom cloud domain name

  • Overcloud node mapping to custom hostnames

  • Controlled node placement

  • Predictable IP addresses

  • Fixed VIP addresses for overcloud endpoints

  • Hyper-converged Infrastructure (HCI)

  • Barbican

  • Octavia

TripleO supports deploying with TLS/SSL on the public OpenStack endpoints. A SSL certificate is configured with the predictable public IP address assigned to the overcloud. A self-signed CA certificate is added to the trusted certificate store on the undercloud and overcloud to validate the contents of the private key.

Red Hat OpenStack Platform 13 also supports the overcloud configuration with TLS everywhere. All communications between overcloud nodes are encrypted with TLS. This setup requires the use of Red Hat IdM as the Certificate Management System.

With DVR, L3 forwarding and NAT are distributed to the compute nodes. This means every compute node acts as a network node for certain types of L3 traffic. DVR can reduce L3 communication network bottlenecks for certain traffic patterns.

2.1. Add Overcloud Domain Records

  1. Add a DNS record to be used as the overcloud’s external domain name with the fixed VIP address:

    [ipa]

4. Enable TLS on Internal Endpoints

You can enable TLS on all overcloud endpoints. Due to the number of certificates required, the director requires integration with Red Hat Identity Management (IdM) that acts as a Certificate Management System (CMS) for the overcloud nodes.

4.1. Enroll Undercloud to IdM

  1. Install`python3-novajoin` package on undercloud:

    [undercloud]
    (undercloud) [stack@undercloud templates]$ sudo dnf install python3-novajoin -y

  2. Replace current dns nameserver on undercloud

    (undercloud) [stack@undercloud ~]$ sudo sed -i 's/192.168.0.1/192.168.0.252/' /etc/resolv.conf

  3. Run the novajoin-ipa-setup command to register the undercloud node and get a One Time Password (OTP):

    [undercloud]
    [stack@undercloud ~]$ sudo /usr/libexec/novajoin-ipa-setup --principal admin --password r3dh4t1\! --server ipa.example.com --realm EXAMPLE.COM --domain example.com --hostname undercloud.example.com --precreate

    Note down the last line, it is the One Time Password to be used in the next step.

  4. Install the crudini package to manipulate the undercloud.conf file:

    [undercloud]
    [stack@undercloud ~]$ sudo yum install -y crudini
    Sample Output
    <<OMITTED>>

5. Enable TLS on Overcloud Public Endpoints

TripleO supports deploying with SSL/TLS on the public and internal OpenStack endpoints. The public VIP of the deployed overcloud needs to be predictable in order for the SSL certificate to be configured properly.

The certificates Common Name can be set to the IP of the public VIP. You have defined predictable IP addresses and VIPs in the ips-from-pool-all.yaml file.

  1. Check the public VIP address defined in the ips-from-pool-all.yaml file:

    [undercloud]
    (undercloud) [stack@undercloud ~]$ grep PublicVirtualFixedIPs ~/templates/ips-from-pool-all.yaml
    Expected Output
      PublicVirtualFixedIPs: [{'ip_address':'10.0.0.150'}]

    • In this lab, the public VIP is 10.0.0.150. The overcloud.example.com DNS record created on the IPA server points to this IP address.

    • If predictable IP addresses are not used, the public VIP is the first address in the external network allocation pool.

5.1. Generate CA and SSL Certificates

In this lab, you create a self-signed CA and use it to validate a self-signed SSL certificate. This is preferable to using a self-signed SSL certificate without a CA. The contents of the private key and certificate files must be provided to the deployment command in Heat environment files. The self-signed CA certificate must also be added to the underclouds trusted certificate store.

The detailed description of this process, as well as instructions for using trusted CA certificates and DNS-based self-signed certificates, can be found in the upstream TripleO documentation.

  1. Copy the enable-tls.yaml file to the templates directory:

    [undercloud]
    [stack@undercloud ~]$ cp ~/rendered/environments/ssl/enable-tls.yaml ~/templates

  2. Copy the inject-trust-anchor.yaml file to the templates directory:

    [undercloud]
    [stack@undercloud ~]$ cp ~/rendered/environments/ssl/inject-trust-anchor.yaml ~/templates/inject-trust-anchor.yaml

  3. Change the relative path in the inject-trust-anchor.yaml template:

    [undercloud]
    [stack@undercloud ~]$ sed -i 's#\.\./\.\.#/usr/share/openstack-tripleo-heat-templates#' ~/templates/inject-trust-anchor.yaml

  4. Check the result:

    [undercloud]
    [stack@undercloud ~]$ grep NodeTLSCAData ~/templates/inject-trust-anchor.yaml
    Expected Output
      OS::TripleO::NodeTLSCAData: /usr/share/openstack-tripleo-heat-templates/puppet/extraconfig/tls/ca-inject.yaml

  5. Generate a private SSL key:

    [undercloud]
    [stack@undercloud ~]$ openssl genrsa -out ~/templates/overcloud-privkey.pem 2048

  6. Create a self-signed CA certificate:

    [undercloud]
    [stack@undercloud ~]$ openssl req -new -x509 -key ~/templates/overcloud-privkey.pem -out ~/templates/overcloud-cacert.pem -days 365 -subj '/C=US/ST=NC/L=Raleigh/O=Red Hat/OU=QE/CN=overcloud.example.com'

  7. Review the contents of the generated X.509 certificate with the following command:

    [undercloud]
    [stack@undercloud ~]$ openssl x509 -in ~/templates/overcloud-cacert.pem -text -noout

  8. Copy the self-signed CA certificate and the IPA CA certificate to the trusted store on the undercloud.

    [undercloud]
    [stack@undercloud ~]$ cat ~/templates/overcloud-cacert.pem /etc/ipa/ca.crt  > ~/cacert.pem
[stack@undercloud ~]$ sudo cp ~/cacert.pem /etc/pki/ca-trust/source/anchors/ca.crt.pem

  9. Update the local CA store to include the new CA certificate:

    [undercloud]
    [stack@undercloud ~]$ sudo update-ca-trust extract

  10. Add the contents of the CA certificate to enable-tls.yaml under the section titled SSLCertificate:

    [undercloud]
    [stack@undercloud ~]$ cd ~/templates
[stack@undercloud templates]$ sed -i -e '/The contents of your certificate go here/r overcloud-cacert.pem' -e '/The contents of your certificate go here/ d' enable-tls.yaml
[stack@undercloud templates]$ sed -i  -e '/-----BEGIN CERT/,/-----END CERT/{s/^/    /g}' enable-tls.yaml
    Commands are literal, don’t replace the text "The contents of your certificate go here".

  11. Add the contents of the private SSL key to enable-tls.yaml under the section titled SSLKey:

    [undercloud]
    [stack@undercloud templates]$ sed -i -e '/The contents of the private key go here/r overcloud-privkey.pem' -e '/The contents of the private key go here/ d' enable-tls.yaml
[stack@undercloud templates]$ sed -i -e '/-----BEGIN RSA/,/-----END RSA/{s/^/    /g}' enable-tls.yaml
    Commands are literal, don’t replace the text "The contents of your private key go here".

  12. Indicate the path where the CA is (for undercloud and overcloud)

    (undercloud) [stack@undercloud templates]$ sed -i "s#InternalTLSCAFile: ''#InternalTLSCAFile: '/etc/pki/ca-trust/source/anchors/ca.crt.pem'#" enable-tls.yaml

  13. Add the CA certificate to the inject-trust-anchor.yaml template under the section titled SSLRootCertificate:

    [undercloud]
    [stack@undercloud templates]$ sed -i -e '/The contents of your certificate go here/r /home/stack/cacert.pem' -e '/The contents of your certificate go here/ d' inject-trust-anchor.yaml
[stack@undercloud templates]$ sed -i  -e '/-----BEGIN CERT/,/-----END CERT/{s/^/    /g}' inject-trust-anchor.yaml
    Commands are literal, don’t replace the text "The contents of your certificate go here".

    The compute nodes in the lab environment have two disk drives. The second disk (/dev/vdb) is intended to be used for Ceph OSD. In the real environment, it is better to use immutable device names.

7. Configure Node Placement

The default behavior for the director is to randomly select nodes for each role, usually based on their profile tag. However, the director provides the ability to define specific node placement. This is a useful method to assign specific node IDs e.g. controller-0, controller-1, etc.

  1. Create an environment file with the Nova scheduler hints:

    [undercloud]
    [stack@undercloud ~]$ cat > ~/templates/scheduler-hints.yaml <<EOF
parameter_defaults:
  ControllerSchedulerHints:
    'capabilities:node': 'controller-%index%'
  ComputeHCISchedulerHints:
    'capabilities:node': 'compute-%index%'
EOF

    The same approach is possible for each role via these parameters:

    • ControllerSchedulerHints for Controller nodes.

    • ComputeSchedulerHints for Compute nodes.

    • BlockStorageSchedulerHints for Block Storage nodes.

    • ObjectStorageSchedulerHints for Object Storage nodes.

    • CephStorageSchedulerHints for Ceph Storage nodes.

    • [ROLE]SchedulerHints for custom roles. Replace [ROLE] with the role name.

    Node placement takes priority over profile matching.

10. Prepare Overcloud for Tenants

As a cloud administrator, you need to perform few post-installation steps to prepare your overcloud for the use by the cloud tenants.

10.1. Create Flavors

Red Hat OpenStack Platform 13 director does not define any flavors in the overcloud automatically. You should create them after the deployment with parameters suitable for the expected workloads.

  1. Create the m1.tiny flavor that will be used for evaluation:

    [undercloud]
    (overcloud) [stack@undercloud ~]$ openstack flavor create --ram 512 --vcpus 1 m1.tiny

10.2. Upload Disk Images

To prepare for the evaluation of the overcloud’s advanced features like distributed virtual routing, key management and load balancing, you need to upload and make public some disk images that can be used by the cloud tenants to launch virtual server instances. Because you deployed overcloud with Ceph as a storage backend for OpenStack services the disk images must be converted to the raw format. Steps below show how to upload the cirros image but you can use similar steps to upload Red Hat Enterprise Linux or other images.

  1. As the overcloud admin user, upload the cirros image to the overcloud’s Glance service:

    [undercloud]
    (overcloud) [stack@undercloud ~]$ openstack image create cirros --disk-format raw --container-format bare --public  --file cirros-0.4.0-x86_64-disk.raw

10.3. Create External Networks

As a cloud administrator, you need to define external networks in your overcloud that can be used by the cloud tenants. In the lab environment, you configured your overcloud with one network providing external connectivity. T

  1. Create the external network in your overcloud:

    [undercloud]
    (overcloud) [stack@undercloud ~]$ openstack network create public \
  --external --provider-physical-network datacentre \
  --provider-network-type vlan --provider-segment 10

  2. Create a subnet on the external network with IP ranges matching parameters of the lab environment:

    [undercloud]
    (overcloud) [stack@undercloud ~]$ openstack subnet create public-subnet \
  --no-dhcp --network public --subnet-range 10.0.0.0/24 \
  --allocation-pool start=10.0.0.100,end=10.0.0.200  \
  --gateway 10.0.0.1 --dns-nameserver 8.8.8.8

13.3. Encrypt Cinder volumes

You can use barbican to manage your Block Storage (cinder) encryption keys. This configuration uses LUKS to encrypt the disks attached to your instances, including boot disks. Key management is transparent to the user; when you create a new volume using luks as the encryption type, cinder generates a symmetric key secret for the volume and stores it in barbican. When booting the instance (or attaching an encrypted volume), nova retrieves the key from barbican and stores the secret locally as a Libvirt secret on the Compute node.

15. Octavia Load Balancer

15.1. Introduction

Octavia is an open source, operator-scale load balancing solution designed to work with OpenStack. Octavia accomplishes its delivery of load balancing services by managing a fleet of virtual machines, containers, or bare metal serverscollectively known as amphorae which it spins up on demand. This on-demand, horizontal scaling feature differentiates Octavia from other load balancing solutions, thereby making Octavia truly suited for the cloud.

Services
ServiceDescription

octavia-worker

Takes sanitized API commands from the API controller and performs the actions necessary to fulfill the API request.

octavia-api

It takes API requests, performs simple sanitizing on them, and ships them off to the controller worker over the Oslo messaging bus.

octavia-health-manager

Monitors individual amphorae to ensure they are up and running, and otherwise healthy. It also handles failover events if amphorae fail unexpectedly.

octavia-housekeeping

Cleans up stale (deleted) database records, manages the spares pool, and manages amphora certificate rotation.
